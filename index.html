Regular Expressions

What is a Regular Expression?

A regular expression (regex) is a pattern that describes a set of strings. Think of it as a language for describing languages. Every regex corresponds to a Finite State Machine (FSM).

Topic 2

Finite State Machines

FSM Components

  • Q — Finite set of states
  • Σ — Alphabet (set of input symbols)
  • δ — Transition function
  • q₀ — Start state
  • F — Set of accepting (final) states

DFA vs NFA

  • DFA: Each state has exactly one transition per symbol. Deterministic.
  • NFA: States can have multiple transitions per symbol, AND epsilon (ε) transitions (free moves without reading input).
  • Both recognize the same languages (regular languages)
  • Every NFA can be converted to a DFA

🔑 Subset Construction — The Algorithm

  1. Start: compute ε-closure(q₀) — all states reachable via ε from start
  2. This set of NFA states becomes the first DFA state
  3. For each DFA state S and each symbol a: compute ε-closure(move(S, a))
  4. This gives you a new DFA state. Add it to the worklist if not seen
  5. A DFA state is accepting if it contains ANY NFA accepting state
  6. Repeat until no new DFA states
Exam tip: On NFA→DFA problems, always show your epsilon-closure calculations. The exam from Spring 2025 and Fall 2023 both had NFA→DFA worth 10-15 points. A "garbage" (dead/trap) state must be added if no transitions exist.

Practice: NFA → DFA by Hand

Given NFA: States {0,1,2,3}, alphabet {a,b,c}, start=0, accept={3}

Transitions: 0→1 on a, 1→2 on ε, 2→3 on b, 0→3 on a

Step 1: ε-closure({0}) = {0} (no ε from 0)
Step 2: move({0}, a) = {1,3}; ε-closure({1,3}) = {1,2,3} (ε from 1→2)
Step 3: State {1,2,3} is accepting (contains 3)!
Step 4: move({1,2,3}, b) = {3}; ε-closure({3}) = {3} → accepting state

Context-Free Grammars

Why CFGs? Regular Expressions Aren't Enough

Regular expressions can't describe nested or recursive structures. For example, matching balanced parentheses ((())) requires a CFG. Programming languages have recursive syntax — functions inside functions, nested expressions — so they need CFGs.

CFG Components

  • Terminals — actual symbols (a, b, 0, +, etc.)
  • Non-terminals — placeholders like S, E, N
  • Productions — rules like S → aSb | ε
  • Start symbol — usually S

Derivation

A string is "in the language" if you can derive it from the start symbol by repeatedly applying production rules.

S → aS → aaS → aaε → aa

Ambiguity in Grammars

A grammar is ambiguous if a string has more than one parse tree (or leftmost derivation). This is bad for programming languages — the meaning should be unambiguous.

To fix ambiguity: add precedence rules (higher-precedence operations are deeper in the tree), or add extra non-terminals to enforce structure.

Spring 2024 asked: Write a CFG for strings accepted by an NFA over {a,b}.

Key insight: Each NFA state → one non-terminal. Transitions → productions. Accepting states → ε production.

State 0 →start: S₀ → aS₁ | bS₀ | ...
Accepting state: S₂ → ε | ...

OCaml Fundamentals

Key Properties of OCaml

  • Statically typed — types checked at compile time
  • Type inference — compiler infers types (NOT dynamically typed!)
  • Immutable by default — variables can't be changed after binding
  • Functional — functions are first-class values
  • Compiled — uses ocamlc or dune
  • NOT dynamically typed despite type inference!

  • FALSE: "OCaml is dynamically typed because it uses type inference" — Inference ≠ Dynamic typing
  • FALSE: "Anonymous functions can't be recursive" — (they can with let rec)
  • TRUE: (fun x → x+1) is alpha-equivalent to (fun y → y+1)
  • TRUE: map can always be replaced by fold

Topic 5

Higher-Order Functions

What Makes a Function "Higher Order"?

A higher-order function is one that takes a function as input and/or returns a function as output. In OCaml, functions are first-class values — they can be stored, passed, and returned like any other value.

map — Transform every element

(* type: ('a->'b) -> 'a list -> 'b list *)
let rec map f l = match l with
| [] -> []
| x::xs -> (f x) :: (map f xs)

(* Example: *)
map (fun x -> x*2) [1;2;3]
(* = [2;4;6] *)

fold_left — Accumulate from left

(* type: ('a->'b->'a) -> 'a -> 'b list -> 'a *)
let rec fold_left f a l = match l with
| [] -> a
| x::xs -> fold_left f (f a x) xs

(* Sum: *)
fold_left (+) 0 [1;2;3]
(* = 6 *)

fold_right — Accumulate from right

(* type: ('a->'b->'b) -> 'a list -> 'b -> 'b *)
let rec fold_right f l a = match l with
| [] -> a
| x::xs -> f x (fold_right f xs a)

fold_right (fun x acc -> x :: acc) [1;2;3] [] (* = [1;2;3] — essentially copies the list *)
Exam tip: "Anywhere we use map, we could use fold instead" — TRUE. Fold is more general. map is syntactic sugar.
map f lst = fold_right (fun x acc -> (f x) :: acc) lst []

Topic 6

Lambda Calculus

The Grammar of Lambda Calculus

Lambda Calculus is a tiny but Turing-complete language with just three forms:

E → x              (variable)
E → λx.E         (abstraction: function definition)
E → E E          (application: function call)
E → (E)          (grouping)

Free vs Bound Variables

A variable is bound if it's inside a lambda that has it as a parameter. Otherwise it's free.

(λx.x a) b
green = bound, red = free
In (λx.(λx.xx)x)x(λy.yf)a
Free variables: the outer x, f, a

Alpha Equivalence

Two expressions are alpha-equivalent (α-equivalent) if they differ only in the names of bound variables. They're semantically identical.

(λx.x) ≡ (λy.y) ≡ (λz.z)
All alpha-equivalent!
(λx.xy) is NOT α-equiv to (λy.yy) — free variable changed!

Call-by-Name (Lazy) Evaluation

Reduce the outermost, leftmost redex first. Arguments are NOT evaluated before passing.

(λx.xx)((λy.yy)(λz.zz))
↓ reduce outermost first
((λy.yy)(λz.zz))((λy.yy)(λz.zz))

Call-by-Value (Eager) Evaluation

Reduce the argument first before applying the function. The argument must be a value (lambda) before substituting.

(λx.xx)((λy.yy)(λz.zz))
↓ reduce argument first
(λx.xx)((λz.zz)(λz.zz))
↓ reduce argument again
(λx.xx)(λz.zz)

Lambda Calculus β-Reduction Practice

Practice reducing these expressions — they appear directly on exams:

((λx. (λy. y x)) y)(λx. x b)
1. ((λx.(λy.y x))y)(λx.x b)
2. ← apply outer lambda: substitute y for x
   (λy.y y)(λx.x b)   // 'y' in body is now the free 'y'
3. ← apply: substitute (λx.x b) for y
   (λx.x b)(λx.x b)
4. ← apply: substitute (λx.x b) for x
   (λx.x b) b
5. ← apply: substitute b for x
   b b  (beta normal form)

What is Operational Semantics?

Operational semantics gives meaning to programs by describing how they execute, step by step. It's a formal way to prove what a program evaluates to. We write rules in the form:

Operational Semantics & Evaluation Order

Spring 2024 asked: What does this print given different opsem rules for &&?

(let _ = print_string "a" in false) && (let _ = print_string "b" in true)
  • If left evaluates first: prints "a" (short-circuit: false && anything = false, b never evaluated)
  • If right evaluates first: prints "b" then "a"
  • If both evaluate regardless: prints "ab"
The ORDER in the opsem rule tells you the evaluation order! Left premise = evaluated first.

Topic 8

Type Systems

Static vs Dynamic Typing

  • Static: types checked at compile time (OCaml, Rust, Java)
  • Dynamic: types checked at runtime (Python, Ruby)
  • OCaml has type inference — compiler deduces types for you. This is STILL static typing.

Soundness vs Completeness

  • Sound: If the type checker says it's OK, it WILL run without type errors. (No false negatives)
  • Complete: If a program is correct, the type checker accepts it. (No false positives)
  • Most type systems are sound but NOT complete. Rust is sound but rejects some safe programs.
  • Soundness does NOT imply completeness!

Type Judgment Rules

Reading Type Rules (Typing Context Γ)

Γ (gamma) is a typing context mapping variables to types. We write Γ ⊢ e : t to mean "in context Γ, expression e has type t".

Variable rule:

Γ(x) = t
Γ ⊢ x : t

If expression rule:

Γ ⊢ e1 : bool   Γ ⊢ e2 : t   Γ ⊢ e3 : t
Γ ⊢ if e1 then e2 else e3 : t

Function rule:

Γ, x:t1 ⊢ e : t2
Γ ⊢ (fun x -> e) : t1 -> t2

Application rule:

Γ ⊢ e1 : t1 -> t2   Γ ⊢ e2 : t1
Γ ⊢ e1 e2 : t2

Rust & Ownership

The Three Rules of Ownership

  1. Every value has exactly one owner
  2. There can only be one owner at a time
  3. When the owner goes out of scope, the value is dropped (freed)

Borrowing Rules

  • You can have many immutable references (&T) at once
  • OR exactly one mutable reference (&mut T)
  • NOT both at the same time
  • References must always be valid (no dangling pointers)

Copy vs Move

  • Copy types (primitives: i32, bool, char, f64): assignment copies the value. Old variable still valid.
  • Move types (String, Vec, etc.): assignment MOVES ownership. Old variable becomes invalid.
let y = x; on an i32 → copy (both valid)
let y = x; on a String → move (x invalid!)

Lifetimes — The Concept

Lifetimes ensure references don't outlive the data they point to. The key insight:

fn longest<'a>(s1: &'a str, s2: &'a str) -> &'a str {
  // 'a means: output lives at least as long as shorter of s1, s2
  if s1.len() > s2.len() { s1 } else { s2 }
}
Dangling reference example from Fall 2023: fn append_one(v: &mut Vec<&i32>) { let n = 1; v.push(&n); } — ERROR because n lives on the stack and is freed when the function returns, but the reference would outlive it.

Garbage Collection

Why Garbage Collection?

When we allocate memory on the heap, someone has to free it. Manual management (C/C++) is error-prone (memory leaks, use-after-free). Garbage collection automates memory reclamation. There are two main algorithms tested:

Reference Counting

Each heap object has a counter of how many pointers reference it. When count hits 0, the memory is freed.

Fatal flaw: Cannot handle cycles! If A → B → A, both counts stay ≥ 1 even when unreachable.

Mark and Sweep

Two phases:

  1. Mark: Start from roots (stack), traverse all reachable objects and mark them
  2. Sweep: Go through all heap objects, free any that are NOT marked
Handles cycles! Cyclic structures are freed if no path from any root reaches them.

Lexer (Tokenizer)

Converts raw text into a list of tokens. A token is a meaningful unit: keyword, number, operator, identifier.

"1 + 23" →
[Num(1); Plus; Num(23)]
The lexer does NOT check grammar. It only checks that each word is a valid token.

Parser

Checks that the token list follows the grammar (CFG) and builds an Abstract Syntax Tree (AST).

[Num(1); Plus; Num(23)] →
Add(Num 1, Num 23)
Parse errors = GRAMMATICALLY invalid (right tokens, wrong order)

Property-Based Testing

Unit Testing vs Property-Based Testing

Unit TestingProperty-Based Testing
TestsSpecific input/output pairsProperties that should hold for ALL inputs
CoverageOnly what you thought ofRandom inputs find unexpected edge cases
Writingassert(square(3) == 9)∀x: square(x) ≥ 0

PBT Components

  1. Property (Relation) — A function taking an input and returning bool. Must be true for all valid inputs.
  2. Generator — Generates random test inputs from the domain.
  3. Test runner — Applies the property to each generated input.

Writing a Property

// Property for 'reverse':
// reversing twice gives original
let relation lst =
  reverse(reverse(lst)) = lst

// Property for 'sort':
// sorted list has same length
let relation lst =
  length(sort(lst)) = length(lst)

📝 Exam Pattern: PBT with a Buggy Function

Spring 2025 gave this buggy mapip function:

fn mapip<F: Fn(i32) -> i32>(f: F, v: &mut Vec<i32>) -> Vec<i32> {
  let mut r = v.clone(); // bug: modifies clone, not v
  for mut item in &mut r { *item = f(*item); }
  r
}

Property p: "The Vec should remain the same length after mapip"

Question: Does property p catch the bug?
Answer: NO. The bug is that the original Vec v isn't modified (it modifies r instead), but the LENGTH of both remains the same. A better property: "v should be modified in-place" or "the returned Vec equals v after applying f to each element".

Garbage Collection
Reference Counting:
  - Each object has a ref count
  - Increment when pointer added
  - Decrement when pointer removed
  - Free when count = 0
  ✗ CANNOT handle cycles!

Mark and Sweep:
  Phase 1 (Mark):
    - Start from root set (stack vars)
    - Follow all pointers, mark reachable
  Phase 2 (Sweep):
    - Scan all heap objects
    - Free anything NOT marked
  ✓ CAN handle cycles!

Key question: "Can X clean up cycles?"
  Reference Counting: NO
  Mark and Sweep: YES
FSM / NFA → DFA Algorithm
NFA components: (Q, Σ, δ, q0, F)
DFA: same but δ is total function

ε-closure(S) = S ∪ all states reachable
               by ε-transitions from S

Subset Construction:
  1. Start: ε-closure({q0}) = first DFA state
  2. For each DFA state T, symbol a:
     next = ε-closure(move(T, a))
     where move(T,a) = all states NFA
     goes to from T on 'a'
  3. Add next to worklist if new
  4. DFA state is accepting if it contains
     ANY NFA accepting state
  5. Add dead/trap state if needed

Accepting string: starts in start state,
follows transitions, ends in accept state
Key Concept True/False
ALWAYS TRUE:
✓ Every CFG language ⊇ Regular languages
✓ Turing machines > FSMs in power
✓ map can always be replaced by fold
✓ (fun x->x+1) α-equiv to (fun y->y+1)
✓ In Rust, variables immutable by default
✓ If A<:B and B<:C then A<:C

ALWAYS FALSE:
✗ OCaml is dynamically typed (it's static!)
✗ Every CFG can be represented by NFA
✗ Anonymous functions can't be recursive
✗ Soundness implies completeness
